# traveling-RAG

AI 旅平險專員 - 基於 GraphRAG 的多層條件文件檢索與理賠推理系統
本專案為生成式 AI 應用課程期末報告。核心旨在建立一套「正確性優先」的智能化文件檢索系統，讓使用者在投保旅平險前能徹底釐清疑慮，並擔任專屬的 AI 理賠精算專員。

# 專案核心目標與用途
在傳統保險諮詢情境中，使用者常面臨條款繁雜、難以理解的問題。雖然通用大語言模型（LLM）具備強大對話能力，但直接用於保險領域時存在知識過時與致命幻覺（Hallucination）的缺點。
本專案透過 GraphRAG（圖結構檢索增強生成） 技術，親手處理資料清洗、知識圖譜構建與動態 Cypher 檢索演算法，旨在達成以下目標：
1.消除資訊盲區：在使用者（或其親友）投保旅平險前，協助徹底解答條款疑慮，打造全天候的 AI 保險專員。
2.嚴謹的理賠推理：針對充滿法律術語的合約，進行高精確度的理賠責任判定與金額量化精算。
3.實踐 AI Orchestration（AI 編排能力）：拒絕盲目微調（Fine-tuning）帶來的彈性喪失與高昂成本，透過向量與圖資料庫的混合檢索，兼顧資料私有化、可溯源性（Citations）與 Token 計算效率。

# 核心技術挑戰與應對策略
保險合約的特殊性，大幅考驗了 RAG 系統在解析與分塊的功力：
1.結構化與非結構化的混合:
保險條款充斥著表格（理賠限額比例）、清單（特別不保與除外責任）與長篇長句。專案透過圖資料庫（Neo4j），將實體（如：事故類型、不保事項）與數值關係網狀鏈結，解決傳統 PDF 分塊切碎導致上下文斷裂的問題。
2.高度語意歧義性例如「出發地班機延誤」與「返回台灣（原出發地）行李延誤不予理賠」在合約定義中極其嚴謹。本系統透過精確的關係路徑與模糊比對，精準捕捉這類細微的語意差異。
3.多層條件邏輯推理理賠常有「特定國家 ＋ 住院超過 N 天 ＋ 實支實付上限」等連鎖判定。專案導入「大腦 B：資深理賠精算師」Prompt 架構，強迫 AI 遵循嚴格的拒賠與計算指引，杜絕腦補。

# 系統架構與硬體需求
本系統採 100% 全本地端隱私保護架構運行，確保敏感的保險資料不外流。
1.硬體需求
  GPU (顯示卡)：強烈建議 NVIDIA RTX 3090 / 4090 (24GB VRAM) 或同級產品。原因：本系統同時運行高效能 27B 模型作為作答與裁判。單一模型於 4-bit 量化後需佔用約 16GB VRAM，若硬體資源不足，會導致頻繁的記憶體置換，造成推理速度極度緩慢。
  CPU：Intel Core i7 / AMD Ryzen 7 以上。
  RAM：建議 32GB 以上。
2. 軟體環境與 AI 工具鏈整合
  本專案完整串聯以下前沿工具，實踐全棧 AI 鏈結：資料清洗與結構化：利用 qwen3.6:27b 輔助將繁雜的保險條款 PDF 清洗並轉換為 Neo4j 友善的巢狀 JSON 結構。
  圖資料庫部署：Neo4j Desktop / Neo4j Docker（負責儲存實體網狀關係）。
  模型運行環境：Ollama（負責在地化部署與高速執行開源 LLM）。
  AI 編排框架：LangChain (langchain-neo4j, langchain-ollama) 負責串接知識圖譜、自動生成 Cypher 語法並驅動雙模組思考鏈。
  開發環境：VS Code + Anaconda (Python 3.10+)。

#　驗證機制與對標分析

```python                                                           │

                                │
                         [輸出 1 或 0 寫入 TXT]

本地端自動化評估機制
  主要作答大腦：Qwen3.6:27b（嚴格依據 GraphRAG 檢索出的 Context 進行理賠責任判定與量化計算）。
  獨立裁判大腦：Qwen3.6:27b（負責比對生成答案與 ground_truth 的最終理賠判定與核心理由是否完全相符，進行 0/1 二元分類）。
  自動化報告：系統會自動測試所有情境，計算整體準確率（Accuracy），並將詳細的作答與判定軌跡存於 eval_results.txt。

# 專案設計與優化歷程記錄
資料前處理優化：保險條款合約極度複雜，初期嘗試直接切片導致表格數據錯亂。後期改用 ChatGPT 輔助清洗，定義出高召回率的巢狀結構體。
Cypher 自動生成防呆：針對 GraphCypherQAChain 的 Prompt 進行多次迭代，加入關鍵字動態模糊比對（CONTAINS）。並針對變動長度路徑 -[r*1..2]- 加入防呆限制，嚴禁使用 WHERE type(r) IN [...] 以解決 List 類型帶來的 Type mismatch 錯誤。
硬體與架構效能優化：初期採用 Gemma 與 Qwen 混合模型架構，但在本地端運行容易因兩大模型爭奪 VRAM 導致嚴重的系統延遲。後期統一收斂至單一高效能 Qwen3.6:27b 4-bit 版本，既維持了圖譜推理的強大邏輯，又大幅提升了單題的推理響應速度。

註：本專案為學術研究與學校報告成果，系統提出之理賠分析與判定內容僅供參考，實際理賠仍應以各保險公司官方條款為準。
